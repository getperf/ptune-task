# Ptune Sync Contract Finalization Draft

## Purpose

This document fixes the current migration target for the caller-visible file
contract used by PtuneSync URI execution.

The target is:

- one caller-visible interop directory
- one mutable `request.json`
- one mutable `status.json`
- no public `request_id`
- no public `runs/<request_id>/...` dependency

Internal PtuneSync runtime artifacts may still use private per-operation
directories, but those are outside the public contract.

## 1. Public File Contract

### 1.1 File Set

The caller-visible minimum contract is:

- `interop/request.json`
- `interop/status.json`

Do not introduce additional required public JSON files at this stage.

Command-local artifacts MAY exist when explicitly documented, but they are
optional and MUST NOT be required for normal caller integration.

Examples:

- optional `pull-backup.json`
- optional debug logs

### 1.2 request.json

Required fields:

- `schema_version`
- `request_nonce`
- `command`
- `created_at`
- `status_file`

Optional fields:

- `input_file`
- `options`
- `workspace_id`
- `meta`

Recommended shape:

```json
{
  "schema_version": 1,
  "request_nonce": "20260328T094500123Z-01",
  "command": "push",
  "created_at": "2026-03-28T09:45:00.123Z",
  "status_file": ".../work/interop/status.json",
  "input_file": ".../work/interop/input.json",
  "workspace_id": "vault-main",
  "options": {
    "list": "default",
    "allow_delete": false
  },
  "meta": {
    "source": "obsidian-plugin"
  }
}
```

### 1.3 status.json

Required fields:

- `schema_version`
- `request_nonce`
- `command`
- `phase`
- `status`
- `updated_at`

Optional fields:

- `message`
- `error`
- `data`
- `instance_id`
- `retry_count`

Recommended shape:

```json
{
  "schema_version": 1,
  "request_nonce": "20260328T094500123Z-01",
  "command": "push",
  "phase": "running",
  "status": "running",
  "updated_at": "2026-03-28T09:45:02.441Z",
  "message": "push started",
  "retry_count": 0
}
```

Final success example:

```json
{
  "schema_version": 1,
  "request_nonce": "20260328T094500123Z-01",
  "command": "push",
  "phase": "completed",
  "status": "success",
  "updated_at": "2026-03-28T09:45:10.017Z",
  "message": "push completed",
  "data": {
    "accepted": 12
  }
}
```

Final error example:

```json
{
  "schema_version": 1,
  "request_nonce": "20260328T094500123Z-01",
  "command": "push",
  "phase": "completed",
  "status": "error",
  "updated_at": "2026-03-28T09:45:04.101Z",
  "message": "push failed",
  "error": {
    "code": "request_invalid",
    "message": "input_file was not found"
  }
}
```

### 1.4 Compatibility Rules

- Clients must ignore unknown fields.
- Missing required fields make the payload invalid.
- While a JSON file is being written, parse failure should be treated as a
  temporary unreadable state and retried.
- Atomic write is required for both `request.json` and `status.json`.

## 2. Request Identity

### 2.1 request_nonce

The public logical request key is `request_nonce`.

Recommended properties:

- generated by the caller for each new logical request
- reused for startup retry of the same logical request
- changed when the caller wants a new execution after the previous one finishes

Recommended format:

- `YYYYMMDDTHHMMSSmmmZ-<sequence-or-short-random>`

Example:

- `20260328T094500123Z-01`

`request_id` may still exist internally inside PtuneSync, but it is no longer a
public caller-managed field.

### 2.2 When to Create a New request_nonce

Create a new `request_nonce` for each new logical operation:

- one `auth-login`
- one `auth-status`
- one `pull`
- one `diff`
- one `push`
- one `review`

### 2.3 Retry Reuse Rule

If startup retry occurs before the request is accepted:

- reuse the same `request_nonce`
- reuse the same `request.json`
- reuse the same `status.json`

### 2.4 New Request Rule

If the previous request is already `completed`:

- the caller may overwrite `request.json`
- the caller must write a new `request_nonce`
- PtuneSync must treat that as a new request

## 3. Lifecycle Rules

### 3.1 phase Values

Fix the final `phase` set to:

- `created`
- `accepted`
- `running`
- `completed`

### 3.2 status Values

Fix the final `status` set to:

- `pending`
- `running`
- `success`
- `error`

### 3.3 Valid Combinations

Recommended combinations:

- `created` + `pending`
- `accepted` + `running`
- `running` + `running`
- `completed` + `success`
- `completed` + `error`

### 3.4 Transition Rules

Recommended transition order:

1. caller writes `request.json`
2. caller may write initial `status.json` as `created/pending`
3. URI launch occurs
4. WinUI dispatcher writes `accepted/running`
5. command execution writes `running/running`
6. final state becomes `completed/success` or `completed/error`

### 3.5 Fields Updated on Every status.json Write

Always update:

- `phase`
- `status`
- `updated_at`
- `message` when available

Always keep stable:

- `schema_version`
- `request_nonce`
- `command`

## 4. Retry Contract

### 4.1 Scope

Retry is a startup safety mechanism only.

Retry is allowed only until `accepted` is observed.

### 4.2 Retry Trigger

Retry may happen if:

- `status.json` does not exist yet
- `status.json` exists but remains at `created/pending`
- `accepted` is not observed within the startup timeout

### 4.3 Retry Stop Condition

Stop retry immediately when:

- `phase == accepted`
- `phase == running`
- `phase == completed`

### 4.4 Retry Limits

Recommended defaults:

- startup wait per attempt: `1000-1500 ms`
- poll interval: `500 ms`
- max startup retries: `2-3`
- full command timeout: `90000 ms`

### 4.5 Duplicate Execution Rule

The WinUI side must treat `(status_file, request_nonce)` as the public logical
request identity.

If the same request is retried:

- do not start a second execution if already `accepted` or `running`
- do not restart if already `completed`

If a different `request_nonce` arrives while the current `status.json` still
represents `accepted` or `running`:

- do not start a second execution in the same interop directory
- return or preserve a busy/error status instead

## 5. One-Directory Policy

### 5.1 Directory Layout

Use one caller-visible interop directory:

- `work/interop/request.json`
- `work/interop/status.json`

### 5.2 Concurrency Rule

One caller-visible interop directory supports one active request at a time.

This is intentional.

If callers need parallel execution in the future, they should use separate
interop directories rather than overloading one `status.json`.

### 5.3 Cleanup Behavior

Because the caller-visible contract uses fixed file names:

- caller-side cleanup is minimal
- PtuneSync private cleanup remains internal
- stale public files are overwritten by the next logical request

## 6. Proposed Freeze for Phase 1

The following should be treated as frozen unless a later contract review
changes them intentionally:

- all external commands stay URI-based
- all observable command state stays file-based
- public files are limited to `request.json` and `status.json`
- `request_nonce` is the public logical request key
- public caller contract does not depend on `request_id`
- public caller contract does not depend on `runs/<request_id>/`
- `review` result may be returned in `status.json.data`
- command-local artifacts such as `pull-backup.json` are optional and
  non-required
- startup retry is allowed only before `accepted`

## 7. Recommended Next Step

After this contract is accepted, implementation should proceed in this order:

1. request/status schema implementation
2. caller request file writing and watcher update
3. WinUI dispatcher accepted write and idempotency
4. startup retry validation
5. command-by-command rollout
