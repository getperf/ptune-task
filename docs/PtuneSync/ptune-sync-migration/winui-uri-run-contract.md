# Ptune Sync WinUI URI Run Contract

## 1. Overview

This document defines a proposed run contract for migrating `PtuneSync`
from `ptune-sync-skel` to a WinUI-based implementation while preserving
URI-based integration from Obsidian.

This is a proposal for migration design.

It does NOT replace the current public documents immediately:

- `docs/ptune-sync-skel/cli-contract.md`
- `docs/ptune-sync-skel/uri-contract.md`

Instead, it describes the recommended target contract for the WinUI app.

---

## 2. Design Goals

The contract is designed to satisfy the following constraints:

- Obsidian continues to launch PtuneSync through URI for all commands
- request / response data is exchanged through files, not stdout
- OAuth login remains compatible with redirect-URI based authentication
- retry can be added safely as a startup watchdog
- duplicate execution caused by retry must be prevented
- partial JSON reads / writes must be avoided through atomic file replacement

---

## 3. Recommended Execution Model

External launch method:

- all commands use URI activation

Internal data exchange:

- all commands use file-based request / status exchange

Recommended flow:

```text
Obsidian
  -> write request.json
  -> launch ptune-sync://run?...request_file=...
  -> watch status.json
  -> optional startup retry until accepted
  -> wait for completed
```

Recommended command coverage:

- `auth-login`
- `auth-status`
- `pull`
- `diff`
- `push`
- `review`

---

## 4. Run Directory Layout

Each request SHOULD use its own run directory.

```text
work/
  runs/
    <request_id>/
      request.json
      status.json
```

Optional auxiliary files MAY be added later:

- `pull-backup.json`
- `events.log`
- `debug.log`

However, the minimum recommended contract is:

- `request.json`
- `status.json`

This keeps the contract compact while preserving request / response separation.

`pull-backup.json` is a command-local artifact rather than a required public file.
It is recommended only for successful `pull --include-completed` runs where the
caller wants a short-lived pre-push safety copy in the same run directory.

---

## 5. Why Separate Run Directories

Per-request run directories are recommended for the following reasons:

- avoids collision between repeated launches
- makes retry idempotency manageable
- keeps one request's status isolated from previous runs
- simplifies debugging and failure analysis

The older single shared file pattern:

```text
work/status.json
```

is easier to implement, but is weaker for:

- retries
- duplicate launches
- overlapping requests
- stale result detection

---

## 6. `request_id`

Each run MUST have a unique `request_id`.

Recommended properties:

- generated by the caller before URI launch
- unique across retries of the same logical request
- reused when retrying the same startup attempt

Example:

```text
20260322T080000Z-a1b2c3
```

Rules:

- one logical request = one `request_id`
- retry MUST reuse the same `request_id`
- a new user operation MUST create a new `request_id`

This is required so the WinUI dispatcher can treat retries as the same request
instead of a second command.

---

## 7. `request.json`

`request.json` is immutable input.

It MUST be fully written before URI launch.

Recommended structure:

```json
{
  "schema_version": 1,
  "request_id": "20260322T080000Z-a1b2c3",
  "command": "push",
  "created_at": "2026-03-22T08:00:00Z",
  "workspace": {
    "run_dir": "work/runs/20260322T080000Z-a1b2c3",
    "status_file": "work/runs/20260322T080000Z-a1b2c3/status.json"
  },
  "args": {
    "list": "_Today",
    "allow_delete": false
  },
  "input": {
    "task_json_file": "work/runs/20260322T080000Z-a1b2c3/input.json"
  }
}
```

Required fields:

- `schema_version`
- `request_id`
- `command`
- `created_at`
- `workspace.status_file`

Clients MUST tolerate unknown fields.

---

## 8. `status.json`

`status.json` is mutable output written by PtuneSync.

It is used for:

- startup acknowledgement
- running state
- final success / error result

Recommended structure:

```json
{
  "schema_version": 1,
  "request_id": "20260322T080000Z-a1b2c3",
  "command": "push",
  "phase": "running",
  "status": "running",
  "timestamp": "2026-03-22T08:00:02Z",
  "message": "push started",
  "retry_count": 0,
  "instance_id": "main",
  "data": null,
  "error": null
}
```

Final success example:

```json
{
  "schema_version": 1,
  "request_id": "20260322T080000Z-a1b2c3",
  "command": "push",
  "phase": "completed",
  "status": "success",
  "timestamp": "2026-03-22T08:00:12Z",
  "message": "push completed",
  "retry_count": 1,
  "instance_id": "main",
  "data": {
    "created": 3,
    "updated": 2
  },
  "error": null
}
```

Final error example:

```json
{
  "schema_version": 1,
  "request_id": "20260322T080000Z-a1b2c3",
  "command": "push",
  "phase": "completed",
  "status": "error",
  "timestamp": "2026-03-22T08:00:12Z",
  "message": "authentication failed",
  "retry_count": 1,
  "instance_id": "main",
  "data": null,
  "error": {
    "type": "AUTH_ERROR",
    "message": "refresh token expired"
  }
}
```

## 9. Phase and Status

`phase` and `status` SHOULD be treated as separate concepts.

Recommended `phase` values:

- `created`
- `accepted`
- `running`
- `completed`

Recommended `status` values:

- `pending`
- `running`
- `success`
- `error`

Recommended combinations:

| phase     | status  | Meaning |
|-----------|---------|---------|
| created   | pending | request file created by caller |
| accepted  | running | PtuneSync accepted request |
| running   | running | command execution in progress |
| completed | success | command succeeded |
| completed | error   | command failed |

This split is important because startup retry should depend on `accepted`,
not on final command completion.

---

## 10. URI Format

All commands SHOULD continue to use URI activation.

Recommended format:

```text
ptune-sync://run?request_id=<id>&request_file=<absolute-or-encoded-path>
```

Alternative command-specific format is also acceptable:

```text
ptune-sync://push?request_id=<id>&request_file=...
ptune-sync://auth-login?request_id=<id>&request_file=...
```

Recommendation:

- externally, URI remains the only launch method
- internally, command dispatch uses `request.json`

This keeps Obsidian-side integration uniform.

---

## 11. Startup Retry Policy

Startup retry is recommended as a safety mechanism.

It SHOULD be used only until the request is accepted.

Recommended policy:

1. Obsidian writes `request.json`
2. Obsidian launches URI
3. Obsidian waits for `status.json.phase == accepted`
4. if not accepted within the startup timeout, relaunch URI with the same `request_id`
5. stop retry when accepted is observed
6. after accepted, only wait for `completed`

Recommended limits:

- `startup_retry_max = 3`
- `startup_wait_ms = 1000-1500`
- `poll_interval_ms = 500`
- `completion_timeout_ms = 90000`

Retry MUST NOT happen after:

- `accepted`
- `running`
- `completed`

Retry MUST reuse the same `request_id`.

---

## 12. Dispatcher Idempotency

Because retries reuse the same `request_id`, the WinUI dispatcher MUST be idempotent.

Recommended behavior:

- if request is first seen
  - create / update `status.json` to `accepted`
  - begin execution
- if the same `request_id` is seen again and status is already `accepted` or `running`
  - do not start a second execution
  - keep existing run
- if the same `request_id` is seen again and status is already `completed`
  - do not re-run
  - keep final result

This is the core requirement that makes retry safe.

---

## 13. Atomic Write Rules

Atomic write behavior is strongly recommended.

For both `request.json` and `status.json`:

- write to a temporary file first
- flush and close file handle
- replace target file atomically

Clients MUST assume:

- partially written JSON is invalid
- missing file means "not yet written"
- parse error means "treat as not readable yet" and retry read

This matches the current goal of avoiding stdout / encoding / partial write issues.

---

## 14. Cleanup Policy

Per-request directories require cleanup.

This is acceptable and recommended, but cleanup SHOULD be delayed and best-effort.

Recommended policy:

- only remove runs in `completed`
- only remove runs older than a threshold
- cleanup failure MUST NOT break command execution
- file lock / antivirus / indexer failures SHOULD only emit warnings

Recommended thresholds:

- keep last `N` runs
- or remove runs older than `7` days

Immediate deletion right after completion is NOT recommended.

---

## 15. Recommended Migration Direction

For PtuneSync WinUI migration, the recommended direction is:

- keep URI launch for all commands
- switch all payload exchange to file-based contract
- introduce per-request run directories
- introduce `request_id`
- introduce `accepted` phase
- add limited startup retry
- make dispatcher idempotent by `request_id`

This preserves the current Obsidian-side URI integration pattern while improving
reliability for WinUI activation and file-based execution.
